# Contributing

Thank you for contributing to Full Spectrum Lab.

Before opening a change:

1. identify the owning repository and version line;
2. link the requirement, Schema, RFC or issue being changed;
3. keep protocol semantics, runtime implementation and example content separated;
4. add or update tests and reproducible evidence;
5. do not include credentials, customer identities or proprietary data;
6. state compatibility, migration and known limitations.

Schema changes require a compatibility note and representative positive and negative fixtures. Runtime changes must not silently discard UNKNOWN, reason codes, evidence references or audit fields. Documentation must distinguish planned, implemented, tested and released states.
